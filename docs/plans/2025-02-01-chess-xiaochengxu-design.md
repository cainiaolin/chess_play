## 📋 需求概述

### 项目目标
开发一个微信小程序，支持用户与不同AI模型进行中国象棋对弈，并支持AI观战模式。

### 核心需求

| 需求项 | 说明 |
|--------|------|
| **AI集成** | 云端API集成，支持多种AI模型 |
| **游戏模式** | 人机对战 + AI观战 |
| **模型配置** | 预设模型 + 用户自定义（混合方式） |
| **功能范围** | 对弈 + 悔棋 + 提示 |
| **AI能力** | 完全自主理解中国象棋规则 |

## 🏗️ 系统架构

### 三层架构设计

```
┌─────────────────────────────────────────────────────┐
│                    小程序前端层                       │
│  - 用户交互、棋盘渲染、游戏状态展示                   │
│  - 组件：棋盘、控制面板、模型选择器、设置页面          │
└─────────────────────┬───────────────────────────────┘
                      │ HTTP/WebSocket
                      ▼
┌─────────────────────────────────────────────────────┐
│                    后端服务层                         │
│  - API网关：处理业务逻辑                              │
│  - 游戏引擎：规则校验、状态管理                        │
│  - AI适配器：统一接口对接不同AI服务商                  │
│  - 用户服务：配置管理、对局历史                        │
└─────────────────────┬───────────────────────────────┘
                      │ HTTPS
                      ▼
┌─────────────────────────────────────────────────────┐
│                    AI模型层                           │
│  - OpenAI、文心一言、通义千问、DeepSeek等             │
│  - 统一适配器接口，支持用户自定义端点                  │
└─────────────────────────────────────────────────────┘
```

### 架构优势
- ✅ **可扩展性**：AI适配器模式使新增模型只需实现统一接口
- ✅ **一致性**：后端游戏引擎保证规则一致性
- ✅ **可扩展**：无状态设计便于水平扩展

## 🔄 规则校验策略

### 前后端双重校验

```
┌─────────────────────────────────────────────────────┐
│                      前端                            │
│  1. 即时响应型校验：兵种走法、基本规则               │
│  2. 用户体验：立即反馈，无延迟                       │
│  3. 乐观更新：先更新UI，后端最终确认                 │
└─────────────────┬───────────────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────────────────┐
│                      后端                            │
│  1. 权威校验：完整规则验证，防止作弊                 │
│  2. 状态管理：维护唯一的游戏状态                     │
│  3. 异常处理：拒绝非法走法，回滚状态                 │
└─────────────────────────────────────────────────────┘
```

### 校验分工

| 校验类型 | 前端 | 后端 |
|---------|------|------|
| 基本走法（马走日、象飞田） | ✅ 即时校验 | ✅ 最终确认 |
| 特殊规则（将帅照面、困毙） | ⚠️ 简化版 | ✅ 完整校验 |
| 胜负判定 | 🔄 客户端计算 | ✅ 服务端权威 |

### 数据流

```
用户走子 → 前端校验 → 乐观更新UI → 发送后端 → 后端权威校验 → 确认/回滚
                ↓                           ↓
           即时反馈错误                   持久化状态
```

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐

└──────┬──────┘    └──────┬──────┘    └──────┬──────┘
       │                  │                  │
   ①点击棋子        │                  │
       │─────────→ ②前端校验        │
       │                  │                  │
   ③乐观更新UI        │                  │
       │                  │                  │
   ④发送走法请求    │                  │
       │──────────────────→ ⑤后端权威校验  │
       │                  │                  │
       │                  │───────→ ⑥构建AI请求
       │                  │                  │
       │                  │←─────── ⑦AI返回走法
       │                  │                  │
       │←───────────────── ⑧返回游戏状态    │
```

## ⚠️ 错误处理设计

### AI相关错误

| 场景 | 处理策略 |
|------|---------|
| AI超时（>30秒） | 自动走随机合法步，记录日志 |
| AI返回非法走法 | 重新提示AI，最多3次，失败则随机步 |
| API调用失败 | 切换备用模型或降级为简单算法 |
| API限流 | 指数退避重试，提示用户稍后再试 |
| AI返回格式错误 | 容错解析，提取坐标信息 |

### 游戏规则错误

| 场景 | 处理策略 |
|------|---------|
| 前后端校验不一致 | 回滚UI状态，提示"走法不合法" |
| 用户断线 | 保留对局状态，重连后恢复（30分钟有效期） |
| 并发走法 | 后端基于turn字段校验，拒绝非法回合 |
| 困毙检测 | 后端每回合检测，宣布无步可走方败北 |

### 网络错误

```typescript
// 统一错误响应格式
interface ErrorResponse {
  code: string          // 'AI_TIMEOUT' | 'INVALID_MOVE' | 'RATE_LIMIT'
  message: string       // 用户友好提示
  retryable: boolean    // 是否可重试
  details?: any         // 开发调试信息
}

// 前端错误边界
- 网络请求失败：显示重试按钮
- WebSocket断开：自动重连，显示连接状态
- 渲染错误：降级为简化棋盘视图
```

## 🛠️ 技术栈

### 前端（小程序）

| 类别 | 选择 | 理由 |
|------|------|------|
| 框架 | 微信小程序原生 | 最优性能，官方API支持 |
| 状态管理 | MobX | 轻量级，响应式 |
| UI组件 | 自定义 + Vant Weapp | 棋盘需自定义，UI用组件库 |
| 通信 | wx.request + WebSocket | 官方API，稳定可靠 |

### 后端

| 类别 | 选择 | 理由 |
|------|------|------|
| 运行时 | Node.js 20+ | 异步IO，适合AI调用 |
| 框架 | Express | 成熟稳定，中间件丰富 |
| 实时通信 | Socket.IO | WebSocket封装，自动重连 |
| 数据存储 | Redis | 内存存储，游戏会话高效 |
| 数据库 | SQLite（可选） | 持久化历史记录，轻量级 |

## 📊 核心数据结构

### 棋盘表示

```typescript
// 棋子类型
type PieceType = 'k' | 'a' | 'b' | 'n' | 'r' | 'c' | 'p'
type PieceColor = 'red' | 'black' | null

// 棋盘：10行9列
type Board = (PieceColor | PieceType)[][]

// 空位: [null, null]
// 红马: ['red', 'n']
// 黑炮: ['black', 'c']
```

### 走法表示

```typescript
interface Move {
  from: { x: number; y: number }    // 起点坐标
  to: { x: number; y: number }      // 终点坐标
  piece: PieceType                  // 移动的棋子
  captured?: PieceType              // 吃子（如有）
  timestamp: number                 // 时间戳
}
```

### 游戏会话（Redis存储）

```typescript
interface GameSession {
  id: string
  board: Board
  turn: 'red' | 'black'
  moves: Move[]
  status: GameStatus
  players: {
    red: { type: 'user' | 'ai', model?: string }
    black: { type: 'user' | 'ai', model?: string }
  }
  createdAt: number
  updatedAt: number
}
```

## 🔐 安全考虑

| 风险 | 防护措施 |
|------|---------|
| 用户作弊 | 后端权威校验所有走法 |
| API Key泄露 | 用户自定义Key加密存储，不上传服务器 |
| DDoS攻击 | 请求限流（每用户10次/分钟） |
| 注入攻击 | 输入校验，AI Prompt模板化 |
| 数据泄露 | WebSocket认证，会话隔离 |
